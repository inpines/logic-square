# 04-analysis — How 的分析展開（Analysis）

> 本資料夾呈現的是 **How 層思考之後的分析結果**，  
> 用來回答：  
> **「在既定結構下，世界如何實際運作？」**

⚠️ 本資料夾 **不是流程規格、不是行程需求、不是技術設計**。

---

## 本層在黃金圈中的位置

```text
What → Why → Structure → How（方法）→ Analysis → Implementation
                                  ↑
                             04-analysis 在這裡
````

- `01-golden-circle/how.md`  
    → 定義 **How 應如何被思考**（方法論）
    
- `04-analysis/`  
    → 呈現 **How 思考後的具體分析成果**
    

> **How（方法）只說一次；  
> Analysis（結果）可以反覆修。**

---

## 本層解決的核心問題

- 世界發生了什麼事？
    
- 事物如何在時間中改變？
    
- 哪些行為是「必要的」，而非習慣或方便？
    
- 哪些行為值得被封裝成穩定作業？
    

> 本層的任務是**驗證結構是否能運作**，  
> 而不是替系統找解法。

---

## 本資料夾結構說明

```text
04-analysis/
├─ flow/                      # 世界怎麼動（發生順序）
├─ sequence/                  # 誰對誰互動（責任視角）
├─ activity/                  # 必須做的事（行為分解）
├─ operations.md              # 可封裝的服務作業
└─ user-story-realization/    # 敘事導覽層（對齊使用者故事）
```

---

## 各子目錄的語意邊界（請勿混用）

### `flow/` — 事件與狀態的推進

**回答：世界發生了什麼？接下來發生什麼？**

- 事件順序（Event Flow）
    
- 狀態如何轉換（State Flow）
    
- 分支與匯合的語意
    

❌ 不描述誰呼叫誰  
❌ 不描述操作步驟

---

### `sequence/` — 互動序列

**回答：誰在什麼時候，對誰做了什麼？**

- 跨角色、跨系統互動
    
- 同步 / 非同步關係
    
- 責任交會點
    

❌ 不承擔業務語意  
❌ 不定義狀態生命週期

---

### `activity/` — 必要行為

**回答：為了回應變化，必須完成哪些事？**

- 人工 / 系統 / 外部行為
    
- 行為的目的與責任
    
- 可替換、可重組
    

❌ 不等同 UI 操作  
❌ 不等同流程步驟

---

### `operations.md` — 服務作業

**回答：哪些行為值得被封裝成穩定能力？**

- 清楚責任邊界
    
- 不承擔價值判斷
    
- 可重用、可組合
    

❌ 不包含業務決策  
❌ 不等同 API 清單

---

### `user-story-realization/` — 敘事導覽層

**回答：一個使用者故事，是如何被既有結構承接的？**

- 用故事「走過」既有 Flow / Activity
    
- 對齊 PO / 業務理解
    
- 明確故事的邊界與不涵蓋事項
    

⚠️ 使用者故事 **不是結構來源**，  
只能作為導覽與對齊。

---

## 明確禁止事項（非常重要）

- ❌ 把本層當成 SOP 或作業手冊
    
- ❌ 把時間排程、SLA、Deadline 寫在這裡
    
- ❌ 把技術細節偷渡進 Analysis
    
- ❌ 為了故事完整而新增不必要行為
    

---

## 與其他層的關係

- 本層 **必須回溯到**：
    
    - `02-domain/`（事物、事件、角色）
        
    - `01-golden-circle/how.md`（分析方法）
        
- 本層 **不得反推**：
    
    - 結構定義
        
    - 需求範圍
        
    - 技術選型
        

---

## 進入 Implementation 的門檻（Gate）

在進入 `05-implementation/` 前，請確認：

-  Flow、Sequence、Activity 彼此一致
    
-  所有行為皆可回溯到事件或狀態變化
    
-  無作業承擔業務判斷
    
-  使用者故事未污染結構
    

---

## 一句定錨

> **Analysis 不是設計系統，  
> 而是驗證這個世界是否真的能這樣運作。**

---

## 命名原則：

- `scenario` 指「一個世界情境」，不是功能
    
- 同一個 scenario 可以同時出現在 flow / sequence / activity

## 一個總原則（請記住）

> **04-analysis 的每一個 `md`，  
> 都應該是在回答：  
> 「如果世界真的這樣運作，結構是否站得住？」**

如果某個分析文件：

- 看起來像 SOP
    
- 看起來像 API 設計
    
- 或開始補救需求模糊
    

那它就已經越界了。
